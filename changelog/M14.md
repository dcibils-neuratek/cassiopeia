# Milestone 14 — Engine expressiveness

Shipped 2026-07-20

More ways to compose and scale work: reuse processes, fan out over collections,
and act when an SLA is blown.

## Functional
- **Sub-processes (call-activity)**: a **Subprocess** node runs another published
  process to completion and merges its result back — compose and reuse flows. The
  sub-process runs in-process and its own instance is recorded for audit. (It must
  be fully automated — no human tasks inside.)
- **Multi-instance (fan-out)**: a service task or subprocess can **run once per
  item** of a context collection, collecting the outputs into a result array —
  e.g. "score each applicant". Configure a collection path, item key, and result
  path.
- **SLA escalation**: when a human task blows its due date, the scheduler
  **escalates** it — bumps it to **high priority** and records a `task.escalated`
  event — so overdue work surfaces to the top of the Inbox automatically.

> Note: true concurrent multi-token gateways (AND-split/join running branches at
> the same time) remain a deeper model change and are still future; multi-instance
> covers the common "do X for each item" fan-out need.

## Technical
- **model** — `SubprocessNode` (+ `NodeType`/`Node` union); `MultiInstance`
  (`collectionPath` / `itemKey` / `resultPath`) on service tasks and subprocesses.
- **engine** — `EngineDeps.runSubprocess?`; subprocess case (single or fan-out);
  service-task case restructured to support fan-out (a `runMulti` helper) while
  keeping retry/error-routing/async; validation for subprocess; new events
  `subprocess.completed`, `task.escalated`.
- **db** — `tasks.escalated` column (migration); `listOverdueTasks` /
  `escalateTask`.
- **runtime** — `runSubprocess()` (persists a child instance, requires synchronous
  completion) injected into `depsFor`; `fireEscalations()` wired into the
  scheduler alongside timers and schedules.
- **web** — Designer subprocess node (palette / colour / render / process-picker
  inspector) and a reusable **Fan-out** editor on both service tasks and
  subprocesses.

## Verified
- Pure-engine unit tests: multi-instance collects one result per item
  (`[1,2,3] → 3 results`); subprocess merges the child's context (`score:720`).
- Live API: parent completes with the child's output merged
  (`subprocess.completed`); a multi-instance service task over a 3-item collection
  produces a 3-element `results` array; an overdue task escalates `low → high`
  with a `task.escalated` event.
- Browser: subprocess palette + inspector + fan-out toggle render; app healthy.
