# Milestone 15 — Notifications & collaboration

Shipped 2026-07-20

Keep people in the loop and let them work together on running instances.

## Functional
- **In-app notifications**: a bell in the top bar with an unread badge and a
  dropdown. Users are notified when a task is **assigned** to them, when a task
  they own is **escalated** (SLA breach), and when a task is **reassigned** to
  them. Opening the panel marks them read.
- **Comments**: every instance has a comment thread (Monitor → detail). Post a
  note; each comment is stamped with author + time — a lightweight audit of human
  discussion alongside the machine audit trail.
- **Reassign / delegate**: from the Inbox, the current assignee (or an admin) can
  **reassign** a task to another user, who is notified.

## Technical
- **db.ts** — `notifications` and `comments` tables; `addNotification` /
  `listNotifications` / `unreadCount` / `markAllNotificationsRead`; `addComment` /
  `listComments`.
- **runtime.ts** — emits notifications on task assignment (in `onParked`) and on
  SLA escalation (in `fireEscalations`).
- **server.ts** — `GET /notifications` (+ unread count) / `POST
  /notifications/read`; `GET|POST /instances/:id/comments`; `POST
  /tasks/:id/reassign` (assignee or admin only; notifies + audits).
- **web** — `NotificationBell` in the App top bar (polls, badge, dropdown, mark
  read); Monitor detail comment thread (list + post); Inbox reassign control.

> Email is not wired to a provider yet — in-app notifications are the delivery
> channel; `addNotification` is the natural hook point for an email/webhook fan-out
> later.

## Verified
- Live API: assigning a task to `admin` creates a notification
  ("Task … was assigned to you", unread 1); posting a comment stores it with the
  author; reassigning `admin → olivia` returns ok and creates
  olivia's notification ("Task reassigned to you by admin").
- Browser: the bell shows a "1" badge and its panel lists the assignment
  notification.
