# Milestone 6 — Resilience & error handling

Shipped 2026-07-01 · commit `4ff9a61`

Make runs survive flaky connectors and turn failures into something operators can
act on.

## Functional
- **Connector retries** with exponential backoff and a per-attempt **timeout**,
  configured per service task in the designer's new **Reliability** panel
  (retries / backoff ms / timeout ms).
- **Error routing**: a service task can designate an outgoing edge as its
  **On failure** path. When the connector ultimately fails, the error is written
  to the context and the run continues down that edge instead of dying.
- **Failed instances capture a reason.** Monitor shows the failure message and a
  one-click **Retry from failed step** button that re-runs the instance from the
  node it stopped on (useful once a flaky dependency recovers).
- New audit events make the story legible: `service.retried`, `service.failed`,
  `instance.failed`.

## Technical
- **model** — `ServiceTaskNode` gains `retries`, `retryDelayMs`, `timeoutMs`,
  `onErrorEdgeId`; `ProcessInstance` gains `error?`.
- **engine** — `runWithPolicy()` wraps the injected `runConnector` with retry +
  `withTimeout` (Promise race). The service-task case catches the final failure:
  routes to `onErrorEdgeId` (stashing `context.error`) or fails the instance with
  a message. `serviceNext()` picks the single non-error outgoing edge;
  `validateDefinition` allows a service task to have a success edge **plus** an
  error edge. New events added to the `EngineEvent` union.
- **db** — idempotent migration (`addColumnIfMissing`) adds `instances.error`;
  `saveInstance` / `getInstance` persist it.
- **runtime / server** — `retryInstance()` re-advances a failed instance from its
  current node; `POST /instances/:id/retry`.
- **connectors** — `mock-fail` connector (always throws) to exercise the path;
  MCP `auth` headers given an explicit `Record<string,string>` type.
- **web** — Designer service-task **Reliability** section (retries/backoff/timeout
  + On-failure edge picker); Monitor failure box + Retry button.
- **infra** — fixed the `typecheck` gate: root `tsconfig.node.json` + web config
  and `@types/node|react|react-dom`, so `pnpm typecheck` runs clean.

## Verified
- Pure-engine unit tests: always-fail → exhausts retries → `failed` (3 calls, 2
  `service.retried`, `instance.failed`); fail-twice-then-succeed → `completed`;
  error-edge routes to the handler with `context.error` set.
- Live API: failing workflow → `failed` with reason and the expected event
  sequence; swap the connector to a working one → **Retry** → `completed`.
- Browser: Monitor shows the failure + Retry and the audit trail; Designer shows
  the Reliability panel on a selected service task.
